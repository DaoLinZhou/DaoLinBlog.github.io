TCP 四次挥手

“挥手” 的目的是终止连接. “挥手” 由客户端或服务端任意一方执行close触发.

假设由客户端主动close

流程图

第一次挥手: Client 发送一个FIN, 用来关闭Client 到 Server的数据传输, Client 进入FIN_WAIT_1 状态;


第二次挥手: Server收到 FIN 后, 发送一个ACK给Client, 确认序号为收到的序号+1(与SYN相同, 一个FIN占用一个序号), Server 进入 CLOSE_WAIT 状态, 这个状态下, 依然会给Client传输数据;


第三次握手: Server 发送一个FIN, 用来关闭Server 到 Client 的数据传输, Server进入 LAST_ACK状态;


第四次握手: Client 收到 FIN后, Client进入TIME_WAIT状态, 接着发送一个 ACK 给 Server, 确认序号为收到序号+1, Server 进入CLOSED状态, 完成四次挥手;

Client 为什么要等待 2MSL

  1. 确保有足够的时间让对方收到ACK包, 如果被动关闭那端没有收到ACK包, 则会触发重发 Finish 包, 一去一回就是2MLS

  2. 避免新旧连接混淆


为什么需要四次握手才能断开连接

全双工: 同一时间, 服务器可以发送数据给客户端, 客户端也可以发送数据给服务器

正因为是全双工, 发送方和接收方都需要FIN报文和ACK报文, 所有各自挥手2次. 由于有一方式被动的, 看起来就像4次挥手

服务器出现大量CLOSE_WAIT的原因

对方关闭socket连接, 我方忙于读写, 没有及时关闭连接

  • 检查代码, 特别是释放资源的代码 (代码有bug)
  • 检查配置, 特别是处理请求的线程配置 (配置不合理)
