思路: 使用 Futamura projections

典型的数据库管理系统 (DBMS) 处理 queries 时会分为多个阶段 (Figure 1.1, Traditional Query Engine)

在前端, 一个 parser 会将给定的 SQL query 转为 logical plan, 之后 query optimizer 会重写这个计划让它变得更为高效并给出一个 physical plan 以备评估.

而后端通常是由一个 interpreter (解释器) 在数据上执行 optimized plan. (在存储数据上 一个 operator 一个 operator 地执行并产生一个又一个 record)


从宏观的视角来看, 数据库基本符合教科书对编译器的描述: 解析器, 前端, 优化器 以及 后端, 一个关键的区别就是在最后一步, 传统 DBMS 通常不生成 machine code


在以前这是个好的选择, 因为当时 I/O 是性能瓶颈, 而通过解释器来解释 query plan 提供了可移植的优势, 而速度没有太多降低 , 因此进行 low-level code generation 是不值得的 (System R: relational approach to database management)

但现在 NVM 硬件发展, OLAP 型查询需求增多, 使得很多查询都是 CPU bound, 因此将 query execution plans (QEPs) 转为本地代码这种想法得到复兴 (old idea)

很多商业系统都使用这种技术 (lmpala, Hekaton, Spark SQL …)


学术界很多人认为困难, 但其实没有必要细致到操作寄存器的地步, 只要生成 C/LLVM 代码就够了(其实这也叫编译).

HyPer 使用 programmatic LLVM 并取得出色的性能, 代价是一些低级实现会渗透到引擎代码的很大一部分


Futamura Projections

1970 年 Codd 提出: “data could be profitably represented as high-level relations without explicit reference to a given storage model or traversal strategy”

1971 年 Yoshihiko Futamura 发表论文 “ “Partial Evaluation of Computation Process An approach to a Compiler-Comp”, 他的观点是 编译器和解释器没有本质上的区别, compilation 可以看作是 **specialization** of interpreter

Specializing Interpreters

Futamura 的思想是对 interpreters 进行 specialization.

就像上面的 pow 函数, interpreter 需要两个参数:

  1. Code of the function to interpret
  2. Input data

如果我们有一个 program specializer 或者说 partial evaluator (基于历史原因叫 mix), 那么我们就可以 specialize query 生成 mix, 再用 mix 和 data 计算结果. 这是因为 specialization 避免了 “interpretive overhead” (例如解释器对查询结构调度的开销)

换句话说就是生成了这个查询的 compiled version

这个叫做 first Futamura projection

Structuring Query Evaluators

First Futamura projection 本身不涉及任何代码分析或优化

然而 specialized code 是和 interpreter 有着类似的结构的

因此一个设计的原则是 : By engineering the source interpreter in
a certain way, we can control the shape of the compiled code

下面分析几种常见模型


The Iterator (Volcano) Model

基于统一的 open/next/close 接口, Figure 3a-b 显示了 Query Execution Plan (QEP) 以及 Volcano 的 interface.

执行时从 root 开始向下调用迭代器, 当得到一个值就会 pop up, 由上层检查是否符合要求

因此这种 mode 叫做 pull-based, 每次得到值都会拉上来

尽管这种模型非常直观, 但 function call 会产生大量开销, 人们希望通过编译来消除这些开销

而且 Figure4b 中 select * from Dep where rank < 10 的每一步都要进行 null check, 因为每个 operator 本身处理的都是 dynamic data, 然而实际只有最底层的 operator 进行 null check 就足够了,这也导致了这种方式的性能差



The Data-Centric (Produce/Consume) Model

由 HyPer 引入, 可以得到更好的编译结果, 被广泛使用, 如: LegoBase, DBLAB 和 Spark SQL.

这个模型中, control flow 是反转的

Data 被 push 给 operators (push-base), 而那些 会 materialize tuples 的 operator 叫做 pipeline breaker

Producer 的任务是获取 tuples (scan 或其他接口), 而 Consumer 执行 operation

从 Futamura projection 的视角来看, inter-operator control flow 不依赖 dynamic data, 因此 specialization 可以移除对 operator structure 的 调度, 留下 Figure 4c 这样紧凑的代码.

而且对比 Volcano model, 每个 operator 只操作 valid data, 不需要重复的 null check



Data-Centric Evaluation with Callbacks

但是 data-centric 这个模型不太直观, 因为它把 evaluation 分散到 producer 和 consumer 中了, 我们需要另一套 API 实现相同的 specialization

Figure 5a 就是 push 模型, HashJoin 有 left 和 right 两个 operators

当调用 produce method 时, 会依次调用 left 和 right 的 produce.

首先调用 left.produce 并执行一段时间, 直到到达 scan 或 pipeline breaker. (step 2)

之后调用 consume, 此时会把 record 存入 hash map 中 (step 3)

再之后调用 right.produce 直到到达 scan 或 pipeline breaker (step 4)

再执行 HashJoin.consume (step 5)

consume 是在 produce 中调用的, 而当一个 operator 的 children 比较多时, consume 就会变得更加复杂

http://loopjump.com/architect-query-compiler-revisited/

push模型可以看做DFS访问执行计划树，第一次访问调用produce触发准备数据，第二次回来的时候调用consume触发消费数据。


作者决定使用 callback 来解决这个问题, 提出一个新模型

Data-centric with callbacks model (LB2)

Figure 5b

每一个join branch 都调用不同的 callback, 先调用 left 再调用 right

op.exec(cb) 可以理解为 operator op 会生成一堆 result 并对每个tuple 执行 callback function

  2. 无法进行 cross-operator 优化, 以及 data layout changes 等等

Programmatic Specialization

为了避免上面这些问题, 论文继续将 specialization 继续下推到 operator 的内部成员的数据结构上，特别是 Record 和各种 HashMap 类

这种方法的主要好处是 Figure 6 中的 main engine code 可以保持不变
